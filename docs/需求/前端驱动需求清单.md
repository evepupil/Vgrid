# 前端驱动需求清单（前端倒逼后端）

> **背景**：前端控制台设计已定稿，见 [`docs/design/console.html`](../design/console.html)（可交互 demo）。
> 后端目前只具备其中一部分能力。本文把前端设计**逐条拆成后端需求**，作为实现基准——
> 前端要什么，后端就补什么。
>
> **用法**：每条需求带唯一编号 `FR-x.y`、一句话描述、**前端要的具体字段**、现状标记、涉及模块/端点。
> 实现时以此为验收清单。改前端设计先改这里。
>
> **状态图例**：✅ 已有（现有后端已满足） · 🟡 部分（有基础，需补字段/扩展） · ❌ 缺（完全没有，需新建）
>
> 最后更新：2026-07-07

## 现有后端 API 面速览（实现前的家底）

| 端点 | 现有返回 | 缺口 |
|---|---|---|
| `GET /api/strategies`、`/api/strategies/{name}`（CRUD） | 策略名 + 配置 | 缺状态、关联实例、样本内指标 |
| `POST /api/backtest` | metrics + equity_curve + fills + n_bars | 缺回撤序列、买入持有曲线、阶梯快照 |
| `GET /api/portfolio/summary` | n_instances / total_equity / total_realized_pnl / total_fee | 缺浮动、占用/上限、日变动、按 mode 隔离 |
| `GET /api/portfolio/runners` | 实例列表（含 realized_pnl / open_lots / last_price） | 缺浮动、占用/上限、夏普、回撤、迷你净值、日变动 |
| `GET /api/watchlist`（CRUD） | symbol / name / added_at | 缺现价、涨跌、振幅、网格适配评分、走势 |
| `GET /api/etf/{symbol}/info` | symbol / name | — |
| `GET /api/state?db=` | 单实例 config / snapshot / metrics / fills / equity_curve / fill_marks | 缺阶梯结构、浮动、风控推演、昨收基准 |

**根本缺口**：① 没有「实盘/模拟盘」双模式概念（只有 paper）；② 没有结构化的网格阶梯状态；
③ 没有实时多标的行情推送；④ 没有网格适配评分；⑤ 没有黑天鹅推演；⑥ 没有市场时段状态。

## 1. 全局：实盘 / 模拟盘 双模式（镜像系统）

> 前端顶部有一个全局开关 `实盘 LIVE ⇄ 模拟盘 SIM`，切换后**整套 console 换一套数据**。
> 这是最大的架构改动。

| # | 需求 | 现状 |
|---|---|---|
| **FR-1.1** | 系统提供两套**镜像账户**：`live`（实盘）与 `sim`（模拟盘）。除关注列表外，实例 / 策略部署态 / 持仓 / 成交 / 盈亏 / 净值全部**按 mode 隔离**，互不影响 | ❌ |
| **FR-1.2** | 所有 mode 相关端点接受 `mode=live\|sim` 参数（或路径前缀 `/api/{mode}/...`），默认 `live` | ❌ |
| **FR-1.3** | **关注列表跨 mode 共享**（单一存储，不按 mode 隔离）——见 §10 | 🟡（已有 watchlist，未涉及 mode） |
| **FR-1.4** | `sim` 模式 = 跟随实时行情逐 tick 喂网格引擎、计算得失，**不触发任何真实下单**。复用现有 `paper` 引擎（引擎 `step`/`desired_orders` 已就绪） | 🟡（paper 已有，需接入 mode 框架） |
| **FR-1.5** | `live` 模式 = 真实交易。**当前无真实执行通道**（roadmap 定「先回测+模拟盘」）。近期 `live` 可为只读 / 半自动（到价推送人工下单），执行适配器（QMT / 券商 / 推送）为后续里程碑。**数据模型现在就要支持 live，即便执行层未接** | ❌ |
| **FR-1.6** | 切 mode 不刷新页面即可切换全部数据；后端保证两套数据可独立查询 | ❌（前端已就绪） |

## 4. 网格阶梯状态（核心，最缺的一块）

> 「网格阶梯」是整个界面的主角。后端要吐出一份**结构化的阶梯状态**供前端画工程图。

| # | 需求 · 前端要的字段 | 现状 |
|---|---|---|
| **FR-4.1** | 提供某实例当前的**结构化阶梯**：一串网格线，每条含 `价格`、`类型`、`份额(如有)` | ✅（切1：`build_ladder_view` + `/api/ladder/preview` + `/api/state` 的 `ladder` 字段） |
| **FR-4.2** | 网格线类型枚举：`sell`(上方持仓卖单，带份额) / `buy`(下方空格挂买单) / `capped`(超出资金上限的排队买单) / `ext`(下沿放大区，`depth>0`) | ✅（切1：`kind` 四类 sell/buy/capped/idle + `depth` 标放大区；复用引擎 `desired_orders` 口径） |
| **FR-4.3** | `现价`在阶梯中的位置（用于画琥珀现价线） | ✅（切1：`current_price`，前端定位现价线） |
| **FR-4.4** | `资金上限线`对应的价格（committed 达 cap 时的临界网格线） | ✅（切1：`cap_price` = 最高一条排队格） |
| **FR-4.5** | `下沿放大区`的网格线（`down_spacing_factor` 放大后的价格） | ✅（切1：`depth>0` 的 rung，前端画琥珀虚线 + 放大区标注） |
| **FR-4.6** | 阶梯随实时价更新：现价线位置、跌破下沿后新增的放大区线，可增量刷新 | ❌（需 FR-11.1 实时推流；切2 通电） |
| **FR-4.7** | 区间元信息：`下沿`、`上沿`、`步长`、`格数`、`间距模式` | ✅（切1：视图含 `window_lower/upper` `step` `grid_count` `spacing_mode`） |

## 14. 优先级与落地顺序（建议）

> 原则：先把「模拟盘 mode 完整跑通 + 阶梯可视化 + 已实现/浮动分离」这条主线闭环，
> 因为它不依赖真实执行通道，且是全系统可信度的核心。实盘执行通道单列，最后接。

## 15. 实现计划（竖切执行）

> **怎么切的**：**竖切，不横切**。每一刀从后端穿到前端，把**一个功能真正跑通、能用、能验**，
> 而不是「先堆完所有后端再堆前端」——那样前端长期对着 mock 空转、后端写完没人验。
>
> **排序三原则**：① **依赖**——被别人依赖的先做（实时报价被 KPI/ticker/关注全要 → 早做）；
> ② **价值**——界面主角、诚实性核心先上；③ **避开最硬的基建**——实盘真实下单通道单列最后，
> 前面一律用「先回测 + 模拟盘」跑通。
>

## 附：几条一定要守住的原则（来自设计与前期讨论）

1. **已实现套利利润 与 持仓浮动盈亏 永远分开**，不许合并成一个「总收益」糊弄自己。
2. **回测与实盘/模拟共用同一份策略引擎**——阶梯 / 成交口径必须一致（已由 `strategy` 引擎保证）。
3. **模拟盘绝不触发真实下单**，只按成交价计得失。
4. **红涨绿跌、买盘红卖盘绿**（中国惯例）——后端只回原始数值 + 昨收，颜色由前端定。
5. **资金上限是黑天鹅硬兜底**，占满即停买——风控面板要能一眼看到最坏占用。
