## 发现的问题

### 1. calmar 排序会被"不交易"的配置刷分（中）

**位置**：`src/vgrid/scan/runner.py:57-59`

```python
if m.max_drawdown == 0:
    return _INFINITE
return m.annualized_return / m.max_drawdown
```

只要 `max_drawdown == 0` 就返回极大值 `_INFINITE`，**不看回报**。问题在于：一个完全不交易的配置（ZERO 模式 + 行情始终在网格外，例如强势单边上涨、价格一直高于 `upper_price`，或扫描时把区间挪到了行情之外），引擎一次都不成交 → 权益曲线是一条直线 → 回撤 0、年化 0 → `metric_value("calmar")` 返回 `_INFINITE`，**排到所有真实策略前面，被当成"最优组合"**。

这正好砸在 M3 的核心目标上——"自动找最优参数"，结果一个躺平的配置拿了第一。`test_calmar_zero_drawdown_ranks_first` 那条用例的 row 年化是正的（0.05），本意是"没回撤且赚钱 → 确实牛"；但当前的守卫分不清"赚着钱且无回撤"和"压根没动过"。

**复现**：任一 ZERO 配置在行情不触网格时，`run_scan` 后 `rank(rows, "calmar")` 的第一名就是它，`metric_value` 是 `10^18`。

**建议**：区分两种"回撤为 0"。最简洁的修法——只在 `max_drawdown == 0 且 annualized_return > 0`（或 `n_sells > 0`）时才返回极大值；否则按"没赚到钱"处理（直接返回 `annualized_return`，通常 ≤ 0，自然排到后面）。这样既保留"无回撤上涨 = 优秀"的语义，又堵住躺平刷分。

### 2. 未知的 vary 字段被静默忽略，扫描"假成功"还会在渲染时崩（低-中）

**位置**：`src/vgrid/scan/space.py:57`（`GridConfig.from_dict(data)` 忽略未知键）+ `src/vgrid/scan/markdown.py:55`（`getattr(cfg, key)`）

`GridConfig.from_dict` 只挑认识的键，多出来的键默默丢掉。所以 `vary` 里字段名拼错（例如把 `grid_count` 写成 `grid_counts`、或大小写不对）：`expand()` 照常产出 N 个 config、不报错，但该维度其实根本没变（如果它是唯一 vary 字段，产出的是 N 个**完全相同**的 config）；更糟的是 `render_scan_report` 里 `_cfg_field` 用 `getattr(cfg, key)` 取这个不存在的字段 → `AttributeError` 直接崩。

用户拿到的就是一份有误导性的扫描报告（或一个崩溃），且很难发现是字段名写错了。

**建议**：`ScanSpec.__post_init__` 或 `expand` 里校验 `vary` 的每个键都是 `GridConfig` 的合法字段（可以用 `dataclasses.fields(GridConfig)` 取字段名集合），不认识就抛 `ValueError`。

### 3. fixed 与 vary 字段同名时，fixed 被静默覆盖（低）

**位置**：`src/vgrid/scan/space.py:53-57`

```python
data: dict[str, Any] = dict(self.fixed)
for k, v in zip(keys, combo, strict=True):
    data[k] = v
```

`vary` 的值后写，同名时直接盖掉 `fixed`，不告警。用户如果同一个键在 `fixed` 和 `vary` 里都给了（手滑或改配置时遗留），`fixed` 里那份就被无声丢弃，行为和直觉相反。

**建议**：检测 `fixed` 与 `vary` 的交集，有重叠就抛错或打印告警。

### 4. `--top` 传负数会切出错误的切片（低）

**位置**：`src/vgrid/cli/app.py`（`_cmd_scan` 调 `render_scan_report(ranked, metric, top)`）+ `src/vgrid/scan/markdown.py:19`（`shown = rows[:top_n]`）

`--top` 没校验下界。传 `--top -1` 时：`rows[:-1]` 会**丢掉最后一行**而不是"展示 0 行"，终端还打印"前 -1 组"。属于用户输入，但行为反直觉。

**建议**：`_cmd_scan` 里 `top = max(0, min(args.top, len(ranked)))`，或 argparse 层限定 `--top` 为正整数。

### 5. run_scan 主路径没有测试，长扫描也无进度反馈（低）

**位置**：`tests/test_runner.py`、`src/vgrid/scan/runner.py:42`

`test_runner.py` 全用构造好的 `BacktestMetrics` 测 `rank` / `metric_value`，**没一个用例真正跑 `run_scan`**（注释明说"用构造的 metrics，不跑真实回测"）。所以"每个 config 跑一次 simulate → 收集 metrics"这条主路径，单元测试里是空的，只靠 CLI 手动冒烟覆盖。配合 #1 的 calmar 陷阱——如果有个 run_scan 的小测试，那种"0 成交配置排第一"的异常一眼就能看出来。

另外 `run_scan` 纯串行，组合数大时（几百上千）终端长时间无任何输出，用户不知道是卡了还是在跑。

**建议**：补一个 `run_scan` 的小测试（两三个 config + 几根 bar，断言每组都拿到 metrics、行数等于 config 数）；`run_scan` 里可选地加个进度回调或定期打印。

### 6. vary 候选值列表里的重复值不去重（低）

**位置**：`src/vgrid/scan/space.py:53`

`vary={"grid_count": [4, 4, 6]}` 这种写了重复值的，笛卡尔积照常展开，产出重复的 config，扫描报告里就是重复行，`size` 也虚高。不致命，但白跑且表里出现一模一样的两行会让人困惑。

**建议**：展开前对每个 `vary` 字段的候选值去重（注意只对可哈希值去重；本场景下都是 JSON 原生类型，可哈希）。

## 小结

scan 模块本身设计干净：笛卡尔积展开复用 `GridConfig.from_dict`、组合数上限保护、指标可换、报告标"样本内最优"防过拟合——这些方向都对。最该修的是 **#1（calmar 被躺平配置刷分）**，它直接削弱 M3"找最优参数"的价值，且 reachable。#2（未知 vary 字段静默失败）紧随其后，关乎扫描结果可不可信。其余几条是输入校验和测试覆盖的小补丁。

另外提醒一句承上启下：M4 模拟盘会"用 scan 选出来的参数"跑实盘，所以 #1 如果不修，模拟盘可能拿着一个躺平的"最优"参数上线——风险会传导下去。
